# SQLite Storage Implementation Spec

## Goal
Replace `MemStorage` in `server/storage.ts` with `SqliteStorage` using `better-sqlite3` (synchronous). 
The new class must implement the exact same `IStorage` interface (all ~40 methods) with identical return types.

## Architecture Decision: NO Drizzle ORM
Use **raw better-sqlite3** directly. Drizzle adds complexity we don't need — the existing types are all TypeScript interfaces (not Drizzle schema tables), and the data model uses JSON columns extensively. Raw SQL is simpler here.

## Key Implementation Notes

### JSON Helper Functions
```typescript
function toJSON(val: any): string { return JSON.stringify(val); }
function fromJSON<T>(val: string | null, fallback: T): T {
  if (!val) return fallback;
  try { return JSON.parse(val); } catch { return fallback; }
}
```

### Tracker entries are stored in a SEPARATE table (`tracker_entries`)
When returning a `Tracker` object, you must JOIN tracker_entries and reconstruct the `entries` array. The `getTrackers()` method returns trackers with their entries embedded (matching the MemStorage behavior).

### Habit checkins are stored in a SEPARATE table (`habit_checkins`)
Same pattern — when returning a `Habit`, include all checkins from `habit_checkins` table.

### Obligation payments are stored in a SEPARATE table (`obligation_payments`)
Same pattern — when returning an `Obligation`, include all payments.

### Profile linking behavior
The MemStorage has bidirectional linking: profiles have `linkedTrackers`, `linkedExpenses`, etc. as arrays, and entities have `linkedProfiles` arrays. Both sides must stay in sync. The SQLite version stores these as JSON arrays in the respective tables.
